# Jarvis bot — runbook (по итогам сессии 2026-06-14)

Как устроен ИИ-чат Jarvis после тюнинга и как им управлять.

## Что под капотом
- **Модель:** `Qwen/Qwen3.7-Max` (Together.ai, streaming-only). Задаётся в серверном `.env` → `TOGETHER_MODEL`.
  Бот зовёт Together через `_together_post` (теперь стримит и нормализует ответ — не трогать без причины).
  Фоллбэк: локальная Ollama (`192.168.1.3:11434`, `qwen3.5-uncensored`), затем Gemini.
- **Личность:** в БД, таблица `prompt_versions` (последняя по `created_at` = активная). НЕ в файле.
  Источник истины текста — `docs/identity-jarvis-v8.md` (его и заливаем в БД). Сейчас активна **v22**.
- **Память чата:** авто, `data/chat-summary.md`, переписывается раз в 24ч (строгая планка: только реально
  повторявшиеся внутряки, лимит 8, ротация). Инъектится в промпт с анти-кринж рамкой.

## Команды в Telegram (админ)
- `/prompt` — показать текущую личность; `/promptlog` — история версий; `/rollback <id>` — откат.
- `/setprompt <текст>` — заменить личность (создаёт новую версию, кэш обновляется сразу).
- `/memory` — показать память чата; `/memory_refresh` — пересобрать; `/memory_clear` — обнулить.
- `/mut_reset` — сбросить контекст (бот забывает историю до этого момента).

## Откат, если что-то поехало
- Личность: `/prompt` в ТГ → или `/rollback <id>` на нужную версию (все версии хранятся).
- Модель: вернуть `TOGETHER_MODEL=...` в серверном `.env` + рестарт. (Стриминг в коде безвреден для любой модели.)

## Деплой изменений персоны (если правишь файл, а не через ТГ)
1. Правишь `docs/identity-jarvis-v8.md`.
2. INSERT в `prompt_versions` (dollar-quoting для кириллицы) — см. `scripts/` примеры.
3. `git pull` на сервере (post-merge хук сам делает stop→kill→start; кэш промпта перечитывается на рестарте).
Деплой репозитория: `ssh -i ~/.ssh/mac-max spedymax@192.168.1.35 "cd /home/spedymax/tg-bot && git pull"`.
НЕ добавлять ручной рестарт — хук уже всё делает.

## Петля само-улучшения
- **Что:** ежедневно 13:00 (launchd `com.spedymax.jarvis-loop` на маке Макса) запускается `scripts/jarvis_loop.sh`.
- **Как:** bash тянет сообщения ПОСЛЕ деплоя текущей персоны (только текущий бот), `claude -p` анализирует
  по рубрике, пишет в `docs/loop/decisions.md`, шлёт дайджест Максу в ТГ.
- **Режим B (сейчас):** только предлагает фиксы, НЕ применяет. Ты одобряешь — применяется.
  Через пару недель доверия → режим A (мелкие фиксы авто). Граница: однострочные правки персоны/памяти = авто;
  переписать секцию / код / модель = всегда с твоим тапом.
- **Мак должен бодрствовать в 13:00** (launchd догонит пропущенный запуск при пробуждении).
- Логи прогонов: `.omc/loop-run.log`; решения: `docs/loop/decisions.md`; playbook: `docs/loop/self-improve-loop.md`.

## Тест-сьют (для будущего тюнинга)
`scripts/test_*.py` — прогон наборов сообщений против текущей персоны (читают `docs/identity-jarvis-v8.md`):
- `test_persona_models.py` — бейк-офф моделей; `test_real_replay.py` — реальные запросы из БД;
- `test_batch*.py` — батчи (одиночные/мультитёрн/edge); `test_genz.py`, `test_v1*.py` — проверка конкретных черт;
- `test_memory_gen.py` — прототип регенерации памяти.
Ключ Together — в локальном `.env` (gitignored). macOS python: использовать httpx/curl (urllib без сертов; Together за Cloudflare — нужен браузерный UA).
